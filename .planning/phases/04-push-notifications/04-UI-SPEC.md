# Phase 4 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Note

Phase 4 is primarily a backend and native infrastructure phase (Firebase setup, Edge Functions, pg_cron reminders). The UI surface is **minimal and additive** — no new screens are introduced. Two interaction points require a design contract:

1. **Deferred permission request** — injected into the existing `createSession` and `joinSession` flows inside `SessionsContext`. The native iOS permission dialog is system-controlled; the app contributes only the moment of trigger and any pre-prompt context UI if needed.
2. **Notification status row in Profile** — the existing `profile.notifications` settings row (already translated, not yet wired) shows the user's current push permission state and allows them to navigate to system settings if needed.

All other UI (SessionForm, SessionCard, SessionList, SpotDetail tabs) is **unchanged** from Phase 3.

## Component Inventory

New UI elements introduced in Phase 4 (all others unchanged from Phase 3):

### 1. Notification Status Row (Profile)

**Location:** Profile settings section, below Language row, above Go Premium row (matches existing `profile.notifications` translation key).

**States:**
- `unknown` — permission not yet requested. Row shows bell icon + "Notifications" label + `ChevronRight`. Tapping does nothing visible (permission requested only from session actions, not here). Row is read-only display.
- `granted` — Row shows bell icon + "Notifications" label + a small filled sky-blue dot or "Active" badge (`text-sky-600 bg-sky-50 text-xs font-bold rounded-full px-2`).
- `denied` — Row shows bell icon + "Notifications" label + "Disabled" badge in slate-500 muted + tapping opens system Settings URL (`Capacitor.Plugins.App.openUrl({ url: 'app-settings:' })`).

**Anatomy:**
```
[Bell icon (sky-500)] [Notifications label (font-medium text-slate-700)] ... [status badge or ChevronRight]
```
Row height: 56px (`p-4` = 16px top/bottom padding + content). Matches existing Profile settings rows.

**Icon:** `Bell` from lucide-react (size 20, `text-sky-500` when granted, `text-slate-400` when unknown/denied).

### 2. Permission Inline Banner (deferred trigger)

**Location:** Rendered once inside SessionForm and inside the join flow (SessionCard), **only if** `push_tokens` has no entry for the current user AND this is the first session action.

**NOT a modal.** It is an inline banner within the existing SessionForm card, appearing between the submit button and any error message. It is only visible for the brief moment between user submitting and the native iOS dialog appearing.

**Anatomy:**
```
[Bell icon (16px, sky-500)]  "We'll notify your spots' fans when you create sessions"
```
- Background: `bg-sky-50 rounded-xl p-3`
- Text: `text-sm text-sky-700`
- No dismiss button — it disappears once the native dialog resolves
- Height: auto (wraps at ~2 lines on small screens)

This banner is purely informational context shown for 0.5–1s before iOS takes over with its native dialog. It is not a custom permission dialog.

## Interaction Contracts

### Permission Request Flow (NOTIF-01)

```
User taps "Announce" (createSession) or "Join" (joinSession)
    → ensurePushToken() called in SessionsContext BEFORE DB write
    → If token already registered: skip permission flow entirely
    → If no token:
        → Show inline banner (sky-50 background, ~0.5s) inside SessionForm
        → Call FirebaseMessaging.requestPermissions()
        → iOS native dialog appears (system-controlled)
        → If granted: getToken() → upsert to push_tokens → proceed
        → If denied: proceed silently (no error, no blocking)
    → DB write (createSession / joinSession) always happens regardless of permission outcome
```

Key contract: **Permission denial never blocks the session action.** The session is created/joined regardless of notification permission outcome.

### Notification Status in Profile (NOTIF-01 indicator)

```
Profile mounts
    → NotificationsContext (or inline hook) checks FirebaseMessaging.checkPermissions()
    → Renders appropriate state badge in the Notifications row
    → If denied: row is tappable → Capacitor openUrl('app-settings:')
    → If granted or unknown: row is display-only
```

### Token Rotation (background)

No UI. `FirebaseMessaging.addListener('tokenReceived')` registered in NotificationsContext on mount. Upserts silently.

- All interactive rows meet 44px minimum touch target (`p-4` = 16px × 2 + ~24px content = 56px).
- The permission banner is `role="status"` so screen readers announce it.
- Notification badges use text labels, not color alone.
- System permission dialog is platform-handled (iOS accessibility built-in).
